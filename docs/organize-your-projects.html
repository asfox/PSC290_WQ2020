A consistent and well-organized structure allows effective collaboration with other people, as well as with your future self. Your analyses should be reproducible and your structure should enable that. As we learn how to organize, we often have files like, "project new" and "project new updated" and "project final" and "project final new". This is terrible practice, and can make it near impossible to go back and update your analyses after a few years. Trust me, I know. 

A projects starts from raw data that should never be edited; consider raw data immutable and only edit derived sources. 

 

A short digression: naming and formatting

(Copied from: Titus Brown)

You may have noticed that a lot of Python code looks pretty similar – this is because there’s an “official” style guide for Python, called PEP 8. It’s worth a quick skim, and an occasional deeper read for some sections.

Here are a few tips that will make your code look internally consistent, if you don’t already have a coding style of your own:

  • use four spaces (NOT a tab) for each indentation level;
  • use lowercase, _-separated names for module and function names, e.g.
    my_module;
  • use CapsWord style to name classes, e.g. MySpecialClass;
  • use ‘_’-prefixed names to indicate a “private” variable that should
    not be used outside this module, , e.g. _some_private_variable;

 

Jupyter Notebooks

In this class, we've be using the jupyter notebook. This doesn't mean it's not important to keep organized! Notebooks, like all other things can quickly grow out of control. Ultimately, your success will rely on other people, and your future self, being able to read your own code. 

Here are some of my tips on how to make that easy.

  • Imports should be at the top, in a cell that's separated from the rest of the code. 
  • Hard-coded variables, such as files or numbers, should be defined as variables in ALL CAPS, and separated from the code that uses them. 
  • Don't repeat yourself. Minimize copy and pasted code. If you can avoid it, don't repeat yourself. 
  • Use markdown to describe different steps (you can convert a cell to a markdown cell using "Cmd^m m").

 

Coding for Projects 

As you move forward working on larger coding projects involved in your science, a single jupyter notebook will not be enough. Notebooks are wonderful for working through bits of code, and sharing projects, but the best way to organize your project probably isn't in a single notebook, stored in a single folder with your data. (That doesn't mean you can't still use notebooks; I use notebooks in larger projects all the time.)

Here are two pages that give suggestions about how to organize projects. Both pages provide an overview of how and why to organize your data/project/etc. and provide Github-based ways to copy their structure. Please read through at least one of these. 

  • This example is a little intense, but well explained: https://drivendata.github.io/cookiecutter-data-science
  • This example is a little more sparse, but provides a nice and flexible way to organize projects, with an accompanying git. https://blog.godatadriven.com/how-to-start-a-data-science-project-in-python

 

P.S. In some of my projects, I don't separate folders for "code" and "notebooks". It really depends on what I am doing. :-)
